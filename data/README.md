# Data

Training and evaluation data for entity type prediction on the Ansible Automation Platform (AAP) MCP Server.

## Graph

`graph_snapshot.json` — Typed composition graph derived from 4 AAP OpenAPI specs (Controller, EDA, Galaxy, Gateway). Each tool is modeled as a typed transformation `input_types -> output_types`. Entity types are extracted from request/response schemas; tools that share types create edges in the graph.

- 79 entity types, 1,060 tools, 259 direct (source, target) pairs

## Training data

`training_data.jsonl` — 10,065 examples across 11 query styles. Each example is a `(query, source_type, target_type, style)` tuple.

### Style distribution

| Style | Examples | Description |
|---|---|---|
| paraphrase | 3,831 (38%) | LLM-generated paraphrases from tool descriptions (5 sub-styles: formal, imperative, conversational, question, vague) |
| template | 2,058 (20%) | Deterministic templates: "List all {X}", "Show {X} for this {Y}" |
| multihop | 1,489 (15%) | 2-hop path queries where source and target are not directly connected |
| noisy | 618 (6%) | Conversational: "The patching job has been running for hours, cancel it" |
| question | 496 (5%) | Question form: "How many {X} are there?", "Does this {Y} have any {X}?" |
| hard_negative | 478 (5%) | Confusing pairs: Instance vs InstanceGroup, Job vs JobTemplate |
| synonym | 392 (4%) | Informal terms: "machines" -> Host, "secrets" -> Credential, "capacity pools" -> InstanceGroup |
| anti_self | 280 (3%) | Vague queries that should route to Platform->X, not X->X |
| augment | 258 (3%) | Targeted disambiguation for underrepresented pairs |
| ambiguous | 90 (1%) | Source-confusing queries: Platform vs domain-specific sources |
| targeted | 75 (1%) | Specific examples for persistent error patterns |

### Data construction

The training data was built in layers, each addressing a specific gap:

1. **LLM paraphrases** (3,831) — Generated by prompting an LLM with tool descriptions and type mappings. Provides naturalistic diversity but only covers direct graph edges.
2. **Deterministic templates** (2,058) — Template-based generation from the graph snapshot. Covers all direct (source, target) pairs with structured query patterns.
3. **2-hop examples** (1,489) — Generated for all 251 (source, target) pairs reachable via 2-hop paths but not directly connected. Without these, the model had zero training examples for multihop test queries.
4. **Anti-self-referential** (280) — Vague queries ("Check the credentials", "Is anything running?") labeled as Platform->X. Prevents the model from predicting X->X for underspecified queries.
5. **Hard negatives + synonyms** (870) — Domain-specific confusing pairs and informal terminology mappings.
6. **Noisy/conversational** (618) — Realistic ops scenarios with situational context: CI pipeline failures, LDAP outages, stuck workflows, capacity issues. Written iteratively by diagnosing which conversational patterns the model failed on.
7. **Targeted** (423) — Disambiguation examples for persistent error patterns: source confusion (Platform vs Organization/Inventory/EE), ambiguous queries, and augmentation for underrepresented pairs.

### Key design decisions

- **Self-loop downsampling**: X->X examples (e.g., Credential->Credential) were downsampled to 30% for types where Platform->X also exists. Without this, the model predicted self-referential types for vague queries (e.g., "list credentials" -> Credential->Credential instead of Platform->Credential).
- **Sqrt-scaled class weights** during training correct for remaining source/target imbalance.
- **Zero test/train overlap**: All 122 test queries are filtered out during data construction.
- **Iterative noisy data**: Noisy examples were written in rounds — each round analyzed which specific test queries the model failed on, diagnosed the root cause (source confusion, target distraction, self-referential), and added targeted training examples.

## Test queries

`test_queries.json` — 122 held-out evaluation queries across 6 difficulty categories. Written from representative AAP operator workflows, independently of the graph.

| Category | N | Description |
|---|---|---|
| clean | 35 | Well-structured, unambiguous operator requests |
| multihop | 22 | Require traversing 2+ edges in the composition graph |
| synonym | 20 | Use informal/alternative terminology for AAP concepts |
| ambiguous | 15 | Multiple valid interpretations or source type confusion |
| noisy | 15 | Real-world conversational language with situational context |
| multipath | 15 | Multiple valid solution paths exist |
